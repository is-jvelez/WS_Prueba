## Qué

Tres cambios sobre `IS_WS_PRUEBA`, todos operando sobre la capa de persistencia SQL Server ya existente:

1. **Auditoría de cambios:** registrar un historial de auditoría para cada operación que modifique un registro (`Crear`, `Actualizar`, `Eliminar`, y la reactivación vía `Actualizar` con `activo=true`): qué operación se ejecutó, sobre qué `id`, y en qué momento (UTC). Persistido en tabla nueva vía Flyway. No se expone como operación SOAP consultable — solo almacenamiento.
2. **Unicidad de `nombre`:** restricción de unicidad a nivel de base de datos sobre `Nombre` de registros activos (`Activo=1`), filtrada/parcial para no chocar con inactivos homónimos. `Crear`/`Actualizar` devuelven error funcional (`codigo=001`) en vez de excepción técnica (`codigo=900`) ante la violación.
3. **Fix `fecha_fundacion`:** corregir la pérdida/truncado del componente de hora entre lo que el cliente envía, lo persistido, y lo devuelto. **Causa raíz ya confirmada** (ver "Contrato de datos"): es un formateo de salida en `PruebaCrudService.BuildOutput`, no un problema de persistencia ni de parsing de entrada.

## Por qué

- **Auditoría:** con persistencia real (ya no en memoria), se necesita trazabilidad de qué operación modificó un registro y cuándo, de cara a soporte/compliance.
- **Unicidad de nombre:** un nombre duplicado activo deja de ser un dato efímero de demo y pasa a ser un problema de integridad real; se prefiere que la base de datos garantice la regla, no solo la capa de aplicación.
- **Fix de fecha:** el contrato de entrada ya acepta fecha+hora (`CampoParser.AllowedDateFormats`), pero el dato se trunca en algún punto de la cadena — inconsistencia entre lo que el contrato promete aceptar y lo que se conserva.

## Alcance por componente

### SQL Server (Flyway)
- Antes de escribir la migración del índice único: `behavior-capturer` consulta `SELECT Nombre, COUNT(*) FROM dbo.Prueba WHERE Activo = 1 GROUP BY Nombre HAVING COUNT(*) > 1`. Si devuelve filas, DETENTE y reporta la lista — no migres hasta que un humano decida cómo resolver los duplicados.
- Nueva migración: crear `dbo.PruebaAuditoria` con las columnas de la tabla anterior.
- Nueva migración (o la misma, a decidir): índice único filtrado `UX_Prueba_Nombre_Activo` sobre `dbo.Prueba(Nombre) WHERE Activo = 1`.
- La columna `fecha_fundacion` ya es `DATETIME2` — **no hace falta ninguna migración de cambio de tipo** para este CR.
- Ninguna migración debe alterar ni recalcular datos existentes.

### Domain (`PruebaCrudService`, `CampoParser`)
- `Crear`/`Actualizar`/`Eliminar`: tras una operación exitosa, escribir la fila de auditoría correspondiente (vía el repositorio, no acoplado directamente a SQL desde el dominio).
- `Crear`/`Actualizar`: capturar la excepción/resultado de violación del constraint único y traducirla a `codigo=001` con el mensaje exacto de la tabla anterior, en vez de dejar que se propague como `codigo=900`.
- `BuildOutput` (`Domain/Services/PruebaCrudService.cs:264`): cambiar `record.FechaFundacion.ToString("yyyy-MM-dd", ...)` a `ToString("yyyy-MM-ddTHH:mm:ssZ", ...)` — causa raíz ya confirmada, no requiere investigación adicional de `implementer`.

### Infrastructure (`SqlServerPruebaRepository`)
- `Create`/`Update`/`Delete`: agregar la escritura de auditoría (misma transacción SQL que la operación principal).
- `Create`/`Update`: propagar la violación del índice único de forma identificable (no como una excepción genérica indistinguible de cualquier otro error de SQL).
- El mapeo de `fecha_fundacion` aquí (`SqlDbType.DateTime2` en `Create`/`Update`, `reader.GetDateTime(3)` en `Map`) ya preserva la hora completa — **confirmado que no es la causa raíz**, no requiere cambios para este fix.

## Criterios de aceptación

**Auditoría**
- [ ] Dado que `Crear` se ejecuta exitosamente, cuando termina, entonces existe una fila en `dbo.PruebaAuditoria` con `Operacion='Crear'`, el `PruebaId` correcto y `FechaUtc` en UTC.
- [ ] Dado que `Actualizar` se ejecuta exitosamente (incluyendo reactivación con `activo=true`), cuando termina, entonces existe la fila de auditoría correspondiente.
- [ ] Dado que `Eliminar` se ejecuta exitosamente, cuando termina, entonces existe la fila de auditoría correspondiente.
- [ ] Dado que una operación falla (`codigo=001` o `900`), cuando esto ocurre, entonces **no** se registra fila de auditoría.
- [ ] Dado el WSDL/contrato SOAP actual, cuando se agrega auditoría, entonces `listaCampos`/`listaCamposSalida` no cambian de forma.

**Unicidad de nombre**
- [ ] Dado un registro activo con `Nombre="Empresa XYZ"`, cuando se ejecuta `Crear` con el mismo `Nombre`, entonces responde `codigo=001` con el mensaje de la tabla de contrato de datos y no crea el registro.
- [ ] Dado un registro activo con `Nombre="Empresa XYZ"`, cuando `Actualizar` intenta poner ese mismo `Nombre` en otro registro, entonces responde `codigo=001` y no aplica el cambio.
- [ ] Dado un registro **inactivo** con `Nombre="Empresa XYZ"`, cuando se crea un nuevo registro activo con ese `Nombre`, entonces la operación **tiene éxito**.
- [ ] Dado que se reactiva un registro cuyo `Nombre` colisiona con otro ya activo, cuando esto ocurre, entonces también se rechaza con `codigo=001`.

**Fix de fecha_fundacion**
- [ ] Dado que `Crear` recibe `fecha_fundacion="2020-10-15T14:30:00Z"`, cuando se consulta con `Consultar`, entonces la respuesta conserva `14:30:00`.
- [ ] Dado el mismo caso, cuando se inspecciona la fila en SQL Server directamente, entonces la columna almacena la hora completa.
- [ ] Dado que `Crear` recibe `fecha_fundacion="2020-10-15"` (sin hora), cuando se consulta, entonces se comporta igual que hoy (`00:00:00Z`) — no regresiona.
- [ ] Dado que la causa raíz ya está confirmada en `PruebaCrudService.BuildOutput`, cuando se aplica el fix, entonces el cambio se limita a esa línea de formateo — no se tocan `CampoParser` ni `SqlServerPruebaRepository`, que ya preservan la hora correctamente.

## Estrategia de pruebas (captura de línea base ANTES de tocar nada)

### Golden master / characterization

Antes de aplicar cualquier cambio, `behavior-capturer` debe capturar como línea base lo que **no debe cambiar**:

- **SQL:** estado actual del esquema de `dbo.Prueba` (columnas, tipos, defaults). Insertar un registro de prueba y capturar exactamente qué queda persistido y qué devuelve `Consultar` hoy — **incluyendo el truncado actual de `fecha_fundacion`**, que es precisamente lo que este CR va a cambiar intencionalmente (por eso no es un invariante a preservar, sino el "antes" documentado del fix).
- **Tiempo de respuesta:** medir el tiempo actual (baseline) de `Crear`, `Actualizar`, `Eliminar` con varias repeticiones, **antes** de agregar la escritura de auditoría. Umbral confirmado: la latencia p95 tras el cambio no debe superar el baseline en más de **+20ms** — `regression-tester` usa este número como criterio objetivo de pass/fail, no una impresión subjetiva de "sin impacto perceptible".
- **Comportamiento actual sobre nombres duplicados:** confirmar y documentar que **hoy `Crear`/`Actualizar` con un `Nombre` duplicado tiene éxito** (no hay restricción). Este es un caso donde el comportamiento **cambia intencionalmente** — se documenta como línea base "antes", no como invariante.
- **Contrato SOAP:** capturar el WSDL actual y una respuesta real de cada operación, para confirmar que la forma del XML no cambia tras el CR.

> Regla general: todo lo que este CR no menciona explícitamente como "va a cambiar" debe seguir produciendo exactamente el mismo resultado que en la línea base. Cualquier diferencia no prevista detiene el pipeline (política HITL, punto 2).

### Pruebas nuevas por componente

- **SQL:** tras la migración, `dbo.PruebaAuditoria` existe con las columnas acordadas; el índice único filtrado existe y solo aplica sobre `Activo=1`; un `INSERT` directo que viole la unicidad falla a nivel de motor.
- **Domain/Infrastructure:** tests xUnit (en `IS_WS_PRUEBA.Tests`) para: escritura de auditoría en cada operación exitosa; traducción de la violación de constraint a `codigo=001`; preservación de hora en `fecha_fundacion` end-to-end.
- **Regresión sobre lo existente:** `regression-tester` re-ejecuta contra la línea base todos los casos que el CR **no** toca (ej. `Crear`/`Actualizar` con nombres no duplicados, formatos de fecha sin hora) y confirma resultado idéntico.

### Integración end-to-end
- `Crear` con fecha+hora → verificar fila en SQL con hora completa → `Consultar` y verificar que la respuesta la conserva → verificar fila de auditoría creada.
- Intentar `Crear` con nombre duplicado activo → verificar `codigo=001` → verificar que **no** se creó fila de auditoría (porque la operación falló).

## Casos borde y validaciones

- **Transaccionalidad de la auditoría (resuelto):** misma transacción SQL que la operación principal — prioriza consistencia (si la auditoría falla, la operación se revierte) sobre latencia marginal.
- **Datos existentes que ya violan la unicidad propuesta (resuelto):** `behavior-capturer` verifica antes de migrar (`SELECT ... GROUP BY Nombre HAVING COUNT(*) > 1` sobre activos). Si hay duplicados, el pipeline se detiene y reporta la lista exacta — ninguna limpieza automática, la decisión de cómo resolverlos queda en manos humanas.
- **Concurrencia:** dos `Crear` simultáneos con el mismo `Nombre` — el constraint de base de datos es la defensa real (una validación previa en código sin el constraint es insuficiente ante condición de carrera).

## Fuera de alcance

- Consulta del historial de auditoría vía SOAP (ninguna operación nueva tipo `ConsultarAuditoria`).

## Riesgo / impacto en comportamiento existente

**Riesgo: medio.** No es un cambio puramente aditivo:
- La restricción de unicidad puede rechazar operaciones que hoy tienen éxito (`Crear`/`Actualizar` con nombre duplicado) — esto es intencional, pero confirmar que el negocio acepta ese cambio de comportamiento antes de aprobar.
- El fix de `fecha_fundacion` cambia un valor de salida ya observable por clientes existentes (`Consultar` empezará a devolver hora donde antes devolvía `00:00:00`) — confirmar apetito de riesgo con quien consuma el servicio hoy.

**Dependencia asumida por el CR original:** el texto fuente de este CR asumía que la persistencia real en SQL Server (`SqlServerPruebaRepository` + tabla `dbo.Prueba`) todavía no estaba implementada/desplegada. **Esto ya no aplica**: `Infrastructure/SqlServerPruebaRepository.cs` y la migración `flyway/sql/V1__create_prueba_table.sql` ya existen y están en uso por `IS_WS_PRUEBA.asmx.cs` en este repo — no hay que esperar ningún CR previo para empezar este.

**Los 6 puntos abiertos originales se resolvieron el 2026-07-17** (naming de tabla/índice, auditoría de la reactivación, transaccionalidad, tipo de columna `fecha_fundacion` + causa raíz del truncado, umbral de latencia, y manejo de duplicados preexistentes) — ver "Contrato de datos" y "Casos borde" para el detalle de cada decisión.
